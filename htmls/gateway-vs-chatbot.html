Conduit · gateway versus chatbot

Same chat box. Different architecture.

This is the useful comparison: not “which model chats better,” but where decisions happen. A normal chatbot centers the model. Conduit centers a governed gateway, with the model only inside bounded steps.

chatbot model-centered path conduit gateway-centered path difference control, audit, authorization
Normal chatbot / ChatGPT-style app
The model is the center of interpretation, memory use, tool choice, and answer composition.
User asks a questionConversation text is sent to the model, often with tools described in the prompt.
▼
Model interprets intentThe same reasoning step may decide what the user means and which tool to call.
▼
Tool call happensAuthorization and business rules are usually wrapped around individual tools.
▼
Model writes answerFailure, missing data, or policy decisions may be hard to replay after the fact.
Conduit Gateway
The gateway is the control plane. The model is used, but never owns identity, IDs, policy, or access.
BFF sends request windowGateway receives a fresh, stateless turn plus verified identity.
▼
Gateway classifies and resolvesModel proposes references; code resolves canonical IDs and routes by manifests.
▼
Gateway gates accessJWT, Cerbos, classification, and coverage checks run before agents are called.
▼
Gateway fans out and synthesizesAuthorized agents return data; answer is grounded and every step is traced.

Less model authority
The model helps interpret and summarize, but cannot grant access, invent IDs, or silently choose policy.
More control surface
Manifests, policy, coverage, and trace events are explicit architecture, not prompt instructions.
Enterprise shape
The chat interface stays familiar while the backend behaves like a governed integration gateway.
The difference is not the chat UI. It is whether the model is the system, or the model is one bounded component inside a controlled gateway.
