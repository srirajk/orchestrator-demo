Negative cases

where the scores come down — on purpose

You asked whether queries can push the eval scores down. They can — and the reason they do is the system behaving correctly. These two are the reliable levers.

N1 · GROUNDING + RELEVANCE ↓rm_jane
What's the combined total value across holdings and cash for REL-00042?
→wealth.holdingsservicing.cash_managementwealth.risk_profile· 3 agents
grounding 0.00relevance 0.50safety 1.00
Proves: asked to compute an aggregate the agents didn't return, the synthesizer refuses — it lists the components ("holdings $1.97M, cash $529K") and defers the math. Relevance drops ("does not answer the combined total"); grounding reads 0.00 (the two figures aren't verbatim in a single agent's output). The dashboard number falls — and that's the guardrail working, not a hallucination.
N2 · RELEVANCE ↓rm_jane
What will the Whitman holdings (REL-00042) be worth at the end of next quarter?
→5 relationship agents
grounding 0.80relevance 0.50safety 1.00
Proves: asked to forecast, it declines — the agents return current data, not predictions. Relevance drops ("acknowledges inability to predict future value"); no figure is invented.
Read grounding correctlyGrounding is a strict, deterministic check — every number in the answer must appear verbatim in an agent's output. It's intentionally unforgiving, so a large multi-number answer (a 5-agent overview) dips below 1.00 (M1 = 0.71) even when nothing is fabricated. Read a sub-1.0 grounding on a big answer as "lots of numbers to verify," not "hallucination." The clean signal is the single-answer queries — S1–S4, all 1.00. Deliberately tanking grounding is best-effort; the dependable "score-down" lever is relevance (ask something the agents can't answer), which N1 and N2 both move to 0.50.

The manifests

how a domain is onboarded — JSON, not code

Every behaviour above — which agents exist, what they answer, what clarifies, what's denied — comes from manifest JSON validated against a schema. There is no domain logic in the gateway. Here are two real ones you can share as templates.

Agent manifest · acme.servicing.nav (the selective single-agent from S3)
{
  "agent_id": "acme.servicing.nav",
  "name": "Asset Servicing NAV",
  "description": "Returns net asset value and as-of date for a fund. Keyed by
     fund_id, NOT relationship_id — intentionally NOT selected by relationship-level
     hero prompts, demonstrating the resolver's selectivity.",
  "domain": "asset-servicing",  "sub_domain": "corporate-actions",
  "audience": "segment",  "protocol": "mcp",
  "connection": { "server_url": "http://servicing-mcp:8082/sse",
                   "tool": "get_nav", "transport": "sse" },
  "skills": [{
    "id": "get_nav",
    "examples": [ // ← this vocabulary IS the routing signal
      "latest NAV for fund X", "net asset value of the fund",
      "fund NAV as of today", "what's the NAV on fund 7781" ]
  }],
  "constraints": { "access_mode": "read",
                   "data_classification": "internal", "sla_timeout_ms": 30000 }
}

The examples array is what the router embeds — write distinctive vocabulary and the agent fires alone; write generic relationship phrasing and it joins the fan-out. Selectivity is authored, not coded.

Domain manifest · wealth-management / private-banking (drives CLARIFY + denials)
{
  "sub_domain_id": "private-banking",  "parent_domain": "wealth-management",
  "entity_types": [{
    "key": "relationship_id", "display": "client relationship",
    "id_pattern": "REL-\\d+", "resolve_type": "relationship", "required": true
  }, /* fund_id, period, ticker_references … */ ],

  "required_context": ["relationship_id"],   // ← missing this ⇒ deterministic CLARIFY (C1)

  "denial_messages": {
    "not-in-book": "That client is not in your coverage.",
    "coverage-transferred": "Coverage for that client has been transferred." },

  "messages": {
    "missing_entity_question": "Which client relationship are you asking about? …" },

  "agents": [ "acme.wealth.holdings", "acme.wealth.performance",
              "acme.wealth.risk_profile", "acme.wealth.goal_planning" ]
}

Adding an agent to a domain = one line in agents[]. Adding a whole business line = one new file plus a coverage-service URL. The user-facing copy ("That client is not in your coverage") lives here too — never in gateway Java.
