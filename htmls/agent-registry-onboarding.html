Conduit · agent registry

How a new agent joins the gateway without changing gateway code.

Think of onboarding as three simple moves: define the line of business, register the specialist agent, then let the gateway derive the runtime wiring and Redis routing entries.

Domain = line of business Sub-domain = narrower context Coverage = allowed resources Agent = callable specialist

First, the words in plain English

Domain A line of business the gateway can route and govern. Examples: Wealth Management, Insurance, HR, Asset Servicing.
Sub-domain A smaller area inside a domain with its own required context. Example: Private Banking inside Wealth Management.
Coverage The user's allowed book of business. In wealth, this answers: which client relationships can this user see?
Entity A real business thing mentioned by the user that must become an ID before calling an agent, like “Acme family” becoming REL-123.
Agent A specialist service the gateway can call, such as Performance, Holdings, Settlement, or HR Policy QA.

One concrete example

Domain: Wealth Management Shared rule: use wealth coverage and wealth entity resolution.
Sub-domain: Private Banking Required context: the gateway needs a relationship_id before fan-out.
Agent: Performance Callable specialist: acme.wealth.performance, reached by HTTP OpenAPI.
User asks: “How did the Morgan account do this quarter?” The gateway resolves “Morgan account” to a relationship, checks coverage, then routes to the Performance agent.

What we capture when an agent is onboarded

This is the contract from the business and agent teams. It tells the gateway which line of business exists, what context is required, and how to call the specialist.

Submitted metadata

Business-line manifestDomain = line of business

The domain says “this line of business exists” and points the gateway to shared business services.

Captured fields
domain_id display_name coverage.discover_url coverage.check_url coverage.resolve_url cache_ttl_seconds memory must_preserve memory can_drop
Why capture this?
The gateway stays generic. Wealth, insurance, HR, or asset servicing can each bring their own services without hardcoding the gateway.

Business-line manifestSub-domain = specific context

The sub-domain says what must be understood before the gateway is allowed to call agents in this area.

Captured fields
sub_domain_id parent_domain resource_scoped entity_types required_context clarification_schema denial_messages messages agent ids
Why capture this?
It tells the gateway when to ask a clarification, when to resolve a name to an ID, and which agents belong to this context.

Agent manifestAgent = callable specialist

The agent manifest says who owns the specialist, where it lives, what it can answer, and what limits apply.

Captured fields
agent_id name description version provider domain sub_domain audience protocol connection capabilities skills examples constraints
Why capture this?
The examples make it routeable. The connection makes it callable. The constraints make it governable.

What the gateway derives, then what Redis keeps hot

Submitted metadata is not enough. The gateway validates it, reads the live agent contract, derives runtime fields, and writes two Redis shapes: exact lookup and semantic routing.

Runtime control plane

Gateway adds runtime fields

1. Validate Reject missing required fields, wrong protocol-specific connection shape, or unknown manifest properties.
2. Introspect HTTP reads OpenAPI by operation id. MCP reads the tool schema from tools/list.
3. Derive Add input_schema, output_schema, resolved_connection, indexed, and registered_at.
4. Embed examples Each skill example becomes a vector row so user intent can find the right agent later.

Redis stores two views

Redis Stack

Exact manifest lookup plus semantic routing index.

agent:{id}

JSON

Full derived agent manifest. Runtime fan-out loads this after routing picks an agent.

registry:agents

SET

All registered agent ids. Used by admin list and existence checks.

vec:{id}:{n}

HASH

One row per example phrase: agent id, domain, sub-domain, read/write flag, and embedding bytes.

intent_idx

HNSW

RediSearch index over vec:. Routing can filter read-only agents and narrow by domain.

The important split in one sentence

FilesDomain and sub-domain manifests are loaded at boot by DomainManifestStore and kept in memory.
RedisAgent manifests and routing vectors are stored in Redis because registration and runtime routing need them.
DeregisterDelete agent:{id}, remove from registry:agents, and delete vec:{id}:*.

Why this is different: the gateway is not just calling tools from a prompt. It has a registry control plane: business manifests define the domain, agent manifests define callable specialists, and Redis makes them discoverable at runtime.

Implementation note: the schema includes an A2A connection shape, but the current introspector path supports HTTP and MCP. The registry comment says RedisJSON, while the current code writes serialized JSON through jedis.set.
