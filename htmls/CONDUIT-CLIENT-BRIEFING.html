Conduit: the Client Briefing

A dual-track reference — the full technical depth, and the plain-English way to say each thing to someone non-technical.

Prepared for Sriraj · Meridian bank engagement · read top-to-bottom once, then use the cheat sheet at the end before you walk in.

Every section comes in two layers. The Technical layer is what you actually built. The 🗣 In plain words callout is the sentence you say out loud when the person across the table doesn't write code. Study both. Present whichever the room needs.

Executive summary — memorize this

Conduit is an enterprise AI gateway for a bank. One plain-English question fans out across specialist agents across several business domains, enforces who is allowed to see what, and streams back a single grounded answer — with the entire decision visible live in a "glass-box" panel.

  • The product thesis is "World B": the gateway carries zero embedded domain knowledge. A new business line is onboarded by adding a manifest (JSON) and a coverage-service URL — never by changing gateway code. That is the moat.
  • One question becomes ten steps: classify intent → extract references → resolve to real IDs deterministically → route by meaning → authorize in three layers → fan out in parallel → synthesize one answer from what came back. The LLM proposes; deterministic code disposes.
  • Fabricated identifiers are structurally impossible. The model extracts human references ("the Calderon Trust"); a deterministic lookup turns them into IDs. The model never invents an account number, and never computes a figure — agent data is untrusted input it may only summarize.
  • It degrades gracefully. If a specialist agent fails, its siblings still answer; the reply names what's missing rather than hanging or silently dropping it. Proven live by killing an agent mid-answer.
  • Onboarding is a form, not a release. A new specialist is a manifest (JSON) declaring its domain, connection, read/write mode, and 3+ example questions; the gateway validates it, embeds those examples in-JVM, and files them in a Redis vector index — from which it is instantly routable. No gateway code, ever.
  • The centerpiece story is a hardening arc: a subtle misroute (an in-coverage client denied access) was found only because of the glass-box trace, root-caused to the router using its weakest signal, and fixed across six commits — all World-B-clean, proven on two domains, cold-boot green.

1. What Conduit is

Conduit is an enterprise AI gateway built for a bank we'll call Meridian. A user types one plain-English question — something like "summarize the position for the Calderon Trust" — and that single question fans out across specialist agents spanning multiple business domains (wealth, insurance, asset-servicing, and so on). Along the way it enforces entitlements (is this person allowed to see this?), and it streams back one grounded answer. Crucially, the whole decision is visible live in a "glass-box" panel — you can watch it think.

Architecturally the split is clean. The gateway is a single Java / Spring Boot service. The specialist agents are Python stand-ins that play the role of external domain teams — in a real deployment those would be owned and operated by the wealth desk, the insurance desk, and so on. The gateway never becomes those teams; it orchestrates them.

Think of Conduit as a brilliant receptionist at a giant bank. You walk up and ask one question in ordinary English. The receptionist figures out which specialist departments can answer, checks your badge to make sure you're allowed, sends your question to all the right desks at once, and hands you back a single clean answer — while a screen next to you shows exactly how the decision was made.

1.1 The World-B thesis (non-negotiable)

Technical

The defining constraint of the whole system: the gateway carries zero embedded domain knowledge. There is no wealth logic, no insurance logic, no client names, no ID patterns, no entity-type literals baked into the Java. A new business domain is onboarded by adding manifest JSON plus a coverage-service URL — never by changing gateway Java. If an engineer finds themselves typing a domain name, a client or entity name, an ID pattern (REL-, POL-…), an entity-type literal, or user-facing domain copy into gateway code, that is the signal they've gone wrong — it belongs in a manifest. A deterministic checker (scripts/world-b-check.sh) greps the gateway source for exactly this coupling and must report CRITICAL: 0.

The receptionist knows nothing about the specific bank. Everything they need — which departments exist, what a "client" even means here, who's cleared for what — they read out of a rulebook (we call it the manifest). Want to add a whole new department tomorrow? You don't send the receptionist back to school. You just hand them an updated rulebook.

1.2 Why this is the moat, not the chatbot

Anyone can build a chatbot. Wire a model to a search box and you have something demo-able in a weekend. That is not the hard part and it is not defensible. The hard, valuable, defensible thing is a manifest-interpreter with zero domain knowledge — a gateway that can safely orchestrate real specialist systems, enforce a bank's entitlement rules, and onboard an entirely new business line as configuration rather than a code release. That engineering discipline — the separation of the reasoning machinery from the domain — is the asset.

The chatbot is the easy 10%. The moat is the other 90%: a system where adding a new division is a config change, not a rebuild, where the machine can't make up an account number even if it wanted to, and where every decision is on the record. Competitors can copy the chat box. They can't cheaply copy the discipline underneath it.

2. The decisioning pipeline — the secret sauce

This is the heart of the system: what happens, in order, between the user's question and the answer streaming back. Ten steps. The through-line to hold onto is this — the language model is used only for the fuzzy, human parts (understanding the question, phrasing the answer); every consequential decision (identity, authorization, which real ID) is made by deterministic code. The LLM proposes; code disposes.

2.1 The ten steps

  1. Ingressrequest arrives

    The request comes in from the Chat BFF (backend-for-frontend) as OpenAI-shape SSE, carrying the conversation window (assembled and compacted by the BFF) and a verified JWT that asserts identity, segment, data classification, and coverage. The token is RS256/JWKS and is verified at every hop.

    RULE Authorization is always the JWT. Nothing the LLM says is ever trusted for authorization.

  2. Intent classifyLLM call #1

    Classify what this turn is: FETCH_DATA, FOLLOW_UP, CLARIFY, CHITCHAT, and so on. The classifier's prompt is compiled from the manifest (entity types + domain context) — there are no hardcoded domain strings in it.

  3. Entity extractLLM call #2

    Pull the human references out of the sentence — "the Calderon Trust", "their holdings". The LLM's job ends at the human reference. The LLM never emits an ID.

  4. Deterministic resolvecode, not LLM

    References become real IDs via the domain's coverage service (/entities/resolve). This step is principal-agnostic — resolution is not filtered by the caller's book of business. Fabrication is impossible: an unknown reference resolves to no ID, which forces a clarification rather than a guess.

  5. Clarifydeterministic decision

    Whether to ask a clarifying question is decided in gateway code, deterministically: if the set of extracted entities intersected with the set of required entities is empty (extracted ∩ required = ∅), clarify. This is not an LLM judgment call.

  6. Agent route (semantic)Redis vector search

    Embed the query, then run a KNN search over a Redis RediSearch HNSW vector index built from the agents' manifest example_prompts. A confidence floor decides which agents are in play. Routing is by meaning, driven by Redis + manifests exactly as designed.

  7. Authorize — three layersthe only gate

    (a) Segment gate and (b) data-classification gate — both structural, evaluated by the Cerbos PDP. (c) Coverage gate — data-aware, evaluated by the coverage service against the book-of-business.

    RULE The CHECK is the only gate. All three layers must pass.

  8. Fan-outvirtual threads, parallel

    Dispatch in parallel on virtual threads to the surviving agents (HTTP + MCP, via a ProtocolAdapter; A2A is stubbed behind the same interface). Partial-result tolerant: a failed agent never cancels its siblings. Join to the deadline, harvest the survivors, synthesize from what came back, and state what's missing.

  9. SynthesizeLLM call #3

    The agents' outputs are handed to the model as delimited DATA, never as instructions. The model summarizes; it never computes, never recalls from memory, never invents numbers. Zero fabricated IDs.

  10. EgressSSE + glass-box + OTel

    Stream back OpenAI-shape SSE (role delta, content deltas, [DONE]). Every decision along the way is emitted as a glass-box trace to Redis (24-hour TTL, shown in the UI) and to OTel → Langfuse / Tempo / Loki / Prometheus.

Trace-shape reference (the pipeline as emitted)
# one turn, glass-box trace (abridged)
ingress      jwt=verified segment=wealth class=confidential coverage=book:rm_jane
intent       FETCH_DATA            conf=0.95      (LLM #1, prompt compiled from manifest)
extract      refs=["Calderon Trust"]  (LLM #2 — no IDs emitted)
resolve      "Calderon Trust" -> REL-00099   (coverage svc, principal-agnostic)
clarify      extracted ∩ required != ∅  -> no clarify needed
route        embed -> HNSW KNN          top=wealth-holdings 0.587
authorize    segment=PASS class=PASS coverage(REL-00099 in book:rm_jane)=PASS
fanout       dispatch[wealth-holdings, wealth-performance]  vthreads, deadline=8s
synthesize   agents-as-DATA -> summary  (LLM #3 — summarize only, no compute)
egress       sse role+delta+[DONE]      trace-> redis(24h) + otel

2.2 The plain-words tour of the same ten steps

Routing (step 6) is "matching your question to the right specialist by meaning, not by exact keywords" — so "how's the Trust doing?" and "give me a performance read on the Calderon account" reach the same desk even though they share no words.

The three-layer check (step 7) is three badge-swipes: "Are you the right kind of person (segment)? Are you cleared for this sensitivity (classification)? And is this client actually in your book (coverage)?" All three must be green.

Partial-result (step 8) is "if one specialist is out sick, you still get everyone else's answer — and it tells you plainly which desk didn't respond, instead of pretending or freezing."

The glass-box (step 10) is a flight recorder for every decision — after the fact you can replay exactly why the system did what it did.

3. Agent-metadata ingestion — how the gateway "learns" a specialist

This is the mechanical heart of World B, and it is worth dwelling on: the gateway has never heard of "wealth" or "insurance." Everything it knows about a specialist arrives as a manifest — a standard form the domain team fills out — which the gateway validates, embeds, and files into a searchable directory. From that moment the specialist is routable. No gateway code is touched. Ever. (All fields below are from the actual pinned schema and code.)

3.1 Registration — the two manifests a domain team submits

Technical

Two kinds of JSON, both living in the ./registry folder (mounted read-only into the container at /registry), neither of them gateway code:

a) The agent manifest — registry/manifests/<id>.json

One file per agent, validated against the pinned registry/agent-manifest.schema.json (draft 2020-12, additionalProperties: false — nothing sneaks in). The real declared fields:

FieldWhat it declares
agent_idStable dotted id, e.g. acme.wealth.holdings (pattern-enforced).
name, description, versionHuman name, one-line purpose, semantic version.
provider{ organization, contactEmail? } — who owns the agent.
domainThe business domain, e.g. wealth-management. Not enumerated in the schema — validated at boot against the loaded domain manifests, never a hardcoded list.
audiencesegment (gated by segment + classification) or enterprise (open to all authenticated users — e.g. HR policy Q&A, segment gate skipped).
protocol + connectionWhich adapter invokes it: http (needs openapi_url + operation_id), mcp (server_url + tool + transport), or a2a (agent_card_url). The A2A shape exists in the schema even though its adapter is stubbed — the seam is declared.
capabilities{ streaming, … }.
skills[]Each skill: id, name, description, tags[] (faceted discovery keys), and examples[] — the representative prompts, minimum 3, that get embedded for semantic routing. These are the "example_prompts."
constraintsaccess_mode (read | write — Phase-1 agents must be read; this replaced the old is_mutating boolean), data_classification (public/internal/confidential/confidential-pii), sla_timeout_ms, optional rate_limit.
sub_domain, max_response_tokens (opt)Sub-domain membership (e.g. private-banking); per-agent response truncation before synthesis.

Note the schema's own comment: input/output schemas are derived by the gateway via introspection, not submitted — the domain team declares intent and connection; the gateway figures out the wire shapes itself.

b) The domain manifest — registry/domains/<domain>.json (+ sub-domain files)

One per domain. The real fields (from wealth-management.json): domain_id, display_name, clarify_style (template | composed — the governance knob from fix 5), clarify_tone, a coverage block (discover_url, check_url, resolve_url, cache_ttl_seconds — the linkage to the book-of-business service), and a memory_compaction policy. Entity types live in the sub-domain file (e.g. domains/wealth-management/private-banking.json): entity_types[] (each with key, extract_as, kind, id_pattern like REL-\d+, resolve_type, required), required_context, denial_messages, and user-facing messages. Every domain string the user ever sees — the ID pattern, the "not in your coverage" copy, the clarify wording — lives here, never in Java.

Registering a new specialist is filling out one standard form. The form asks: what's your name, what department are you in, how do we reach you (a web address or a tool), are you allowed to change things or only read, how sensitive is your data, and — most importantly — "give us at least three example questions you can answer." A second form describes the department itself: what its IDs look like, what to say when someone's not allowed in, how warmly to ask follow-up questions. Two forms. Zero code.

3.2 The ingestion pipeline — from a JSON file to a routable agent

Technical

On boot, RegistryBootstrapLoader fires on Spring's ApplicationReadyEvent. It waits for the in-JVM embedding service to be ready, ensures the vector index exists, then scans ${conduit.registry.location}manifests/*.json (which is file:/registry/ in the container) and pushes each file through exactly the same registration pipeline as the live POST /admin/agents endpoint — there is no special bootstrap code path. Invalid manifests are logged and skipped; they never abort startup.

The register() pipeline, per agent, is four steps:

# AgentRegistry.register(submissionNode) — the real pipeline
1. validate     ManifestValidator.validate(node)      // against agent-manifest.schema.json
2. introspect   AgentIntrospector.introspect(...)      // DERIVE input/output schemas + resolved connection
3. persist      HSET/JSON.SET agent:{id} = full manifest   // RedisJSON
                SADD  registry:agents  {id}          // the roster set
4. index        VectorIndex.index(stored)              // embed example prompts -> HNSW

Step 4 is where the magic lives. VectorIndex.index() first removes any existing vectors for that agent_id (a clean re-index on update), then, for every example prompt, embeds it in-JVM (DJL + all-MiniLM-L6-v2, 384-dim — no network hop) and writes one Redis HASH entry:

# one vector entry per example prompt
key   vec:{agent_id}:{example_index}       # dense coverage: each prompt indexed separately
HSET  agent_id     = acme.wealth.holdings
      domain       = wealth-management
      sub_domain   = private-banking
      is_mutating  = 0          # derived: access_mode "read"->0, "write"->1
HSET  embedding    = <384 x float32, little-endian raw bytes>

# the index itself (RediSearch), created once:
FT.CREATE intent_idx ON HASH PREFIX vec:
   agent_id     TAG
   domain       TAG
   sub_domain   TAG
   is_mutating  NUMERIC
   embedding    VECTOR HNSW  DIM 384  TYPE FLOAT32  DISTANCE_METRIC COSINE
                       M 16  EF_CONSTRUCTION 200  INITIAL_CAP 500

At route time (pipeline step 6), search() embeds the query and issues a KNN over intent_idx with the filter @is_mutating:[0 0] (read-only agents only), optionally AND-ed with @domain:{…}, then de-duplicates by agent_id keeping the best similarity (1 − cosine distance). That is the entire mechanism behind "route by meaning."

When the form arrives, the system does four things automatically: checks the form is filled out correctly, figures out the exact data shapes on its own, files the specialist in a directory, and — the clever bit — reads each of those example questions, converts them into a "meaning fingerprint," and pins them on a searchable board. Later, when a real question comes in, the system takes its fingerprint and finds the closest pins. That's how a brand-new specialist becomes findable the instant its form is filed — nobody wrote code to teach the system about it.

3.3 Why the metadata drives everything downstream

The manifest isn't documentation — it is the wiring. Each field powers a specific gate or step:

Manifest fieldDrives…
skills[].examplesSemantic routing. Embedded into intent_idx; the query's nearest neighbours are the candidate agents.
domainThe routing domain-filter (@domain tag) and the segment→domain authorization mapping the Cerbos policy evaluates.
constraints.access_modeRead-only fan-out. Stored as is_mutating; only [0 0] (read) agents are ever eligible to route. Write agents are structurally excluded.
audienceWhether the segment gate applies (segment) or is skipped (enterprise).
constraints.data_classificationThe classification gate — matched against the caller's cleared classification.
domain coverage.{resolve,check,discover}_urlThe coverage/resolve linkage: reference→ID resolution and the data-aware book-of-business CHECK.
protocol + connectionWhich ProtocolAdapter invokes the agent, and how to reach it.

Every line on the form has a job. The example questions decide who gets your question; the department decides which security rules apply; the "read-only" flag guarantees a specialist that can only look things up can never be tricked into changing something; and the coverage web addresses are how the system checks the client is really in your book. Fill the form out well and the system behaves correctly — because the form is the behaviour.

3.4 Onboarding with zero gateway code — the World-B payoff

Putting it together: to add a new agent, drop its manifest into registry/manifests/; to add a whole new business domain, add registry/domains/<domain>.json (plus sub-domain file) and stand up a coverage service at the URL it names. On restart the bootstrap loader validates, introspects, persists and indexes it — and it is immediately routable. No image rebuild (the folder is a mounted volume), no gateway Java change, and scripts/world-b-check.sh stays at CRITICAL 0. Live onboarding is the same pipeline via POST /admin/agents.

Updates and re-ingestion are safe by construction: re-submitting a manifest re-runs the full pipeline, and VectorIndex.index() deletes the agent's old vectors before writing the new ones (a clean re-index, no stale prompts left behind). deregister() removes it from both the registry:agents roster and the vector index.

This is the whole thesis in one motion: a new specialist, or an entire new division, is a file you drop in a folder plus a web address for its records — not a software release. Restart (or call one admin endpoint) and it's live and answering. Change your mind? Edit the form and re-submit; the old fingerprints are wiped and replaced automatically. The receptionist never went back to school; you just updated the directory.

4. Design decisions & the locked stack

3.1 Flat planning, not a planner

Technical

A deliberate scoping decision: build the simple path, leave the seam. The system uses flat plans — not a hierarchical planner or agent-orchestrator — and flat semantic routing. It speaks HTTP and MCP today, with A2A stubbed behind an interface (ProtocolAdapter). The interfaces for the scale version are defined; the scale version itself is intentionally not built yet. This is anti-over-engineering as a discipline: the seams exist so the fancier version can drop in without a rewrite.

We built the straight road that works today and left clean "sockets" to plug the fancier version into later. We deliberately did not gold-plate it with machinery the demo doesn't need. That's a maturity signal, not a gap.

3.2 The stateless gateway

Technical

The gateway holds no server-side carry-forward. There is no per-user memory living inside it. It re-derives everything it needs, every turn, from the conversation window the client sends. This keeps the gateway horizontally trivial to scale and easy to reason about — but, as the hardening arc will show, it also means the router only sees what's in that window, which is exactly where a subtle bug hid.

The receptionist has no memory of their own between questions — they re-read the whole conversation slip you hand them each time. Great for scaling (any receptionist can pick up any conversation), and it's the reason a certain bug was so sneaky: if the slip only shows your last sentence, the receptionist is working with less than they should.

3.3 Custom memory compaction — the BFF owns the conversation

Technical

The gateway is stateless (§3.2), so something else has to remember the conversation — and that something is the Chat BFF. The gateway re-derives everything per turn from the message window it is handed; the BFF is what assembles that window. On every turn, before it calls the gateway, the BFF builds a bounded context window — small enough to stay within a token budget, complete enough that the gateway can still classify and route correctly — and sends that up. This is a purpose-built compaction layer that lives in apps/chat/bff/…/memory, four collaborating classes:

  • ContextAssembler builds each turn's window: the most-recent messages kept verbatim, filled newest-first until one more would exceed the token budget (context.max-tokens) — the latest user message is always included. When the older transcript no longer fits, a rolling summary of it is prepended as a leading system message ("Summary of earlier conversation: …"), and — a deliberate nicety — it is only attached when messages were actually dropped, so a short chat that fits whole never pays for a summary.
  • LlmSummaryService generates that rolling summary on a background virtual thread, fire-and-forget: it never blocks or fails the user's turn, and if the summary LLM is unreachable the previous summary is left untouched. Its prompt summarizes "topics/intent only" and is strictly facts-free — no values or figures, no client / account / policy IDs treated as truth, no entitlement conclusions. The summary is continuity scaffolding for classification, never a data store; the gateway stays the sole ground truth and re-fetches + re-authorizes every turn.
  • TokenCounter sizes the budget with a real in-JVM tiktoken (jtokkit, encoding from config), degrading to a chars/4 heuristic if the tokenizer is unavailable — so context assembly never fails on a token-counting problem.
  • A summaryWatermark on the conversation records how many messages have already been folded into the summary. The assembler asks for a fresh summary only when there is none yet, or when the transcript has grown materially past that watermark — so the summary rolls forward instead of re-summarizing the same history every turn.

One concurrency subtlety is load-bearing. The summary writer and the post-stream "touch / auto-title" writer both run on background threads against the same conversation document. If either persisted the whole entity, it would clobber the other's field — a lost update that (in the code's own words) "silently defeats context compaction." Both writers therefore do field-scoped updates ($set only their own fields, scoped to the owning user) so they can never race on the summary field.

The guardrail. Compaction can fail silently — the worst kind of failure — so it is watched by a metric: chat_compaction_summary_attached_total{attached="true"}, surfaced on the Grafana compaction dashboard. As the code comment states outright, "if attached=true stays 0, compaction is silently broken." If turns are happening and the summary is never being attached, that counter flatlines at zero — the canary that the rolling summary has quietly stopped working.

# one turn, context assembled by the BFF (not the gateway)
transcript   42 msgs / ~9,800 tok       # full history in the BFF's store
budget       max-tokens                 # measured by TokenCounter (jtokkit)
summary      watermark=30  rolling          # 30 older msgs already folded in
assemble     [system] "Summary of earlier conversation: …"  # facts-free, topics/intent only
             + recent msgs VERBATIM, newest-first, within budget
metric       chat_compaction_summary_attached_total{attached="true"} +1   # canary alive
-> gateway   receives the compacted window, re-derives + re-authorizes

Why it mattered to the hardening arc. Because compaction shapes exactly what the gateway sees each turn, the entire multi-turn investigation (§5) had to be run through the real BFF path — isolated gateway calls would have hidden the bug, because they bypass the very window-assembly that was feeding the router its (too-thin) signal. That is precisely why the method insists on "tested through the real multi-turn BFF path, not isolated gateway calls."

Because the receptionist keeps no memory of their own, the front desk (the BFF) keeps the running notes. Every time you ask something, it hands the receptionist a tidy packet: your last few sentences word-for-word, plus a short "here's what we've been talking about" note that stands in for everything older — and that note deliberately holds only topics, never actual figures or account numbers. As the chat gets long it keeps that note rolling instead of re-reading the whole history each time. There's even a warning light — a dashboard counter — that goes dark if the note-taking ever quietly stops. And it's exactly why the big bug hunt had to be run through the real front desk: testing the receptionist alone would have hidden the problem, because the packet is the very thing that was too thin.

5. The multi-turn hardening arc — the centerpiece

This is the story that separates "someone who demoed an AI toy" from "someone who engineered a system." Tell it as a detective story: a symptom, a disciplined investigation that ruled out the easy answer, a precise root cause, and six clean fixes proven twice over.

4.1 The trigger

Screenshots showed a well-specified request for an in-coverage client failing. As relationship manager rm_jane, the request "summarize for Calderon Trust (REL-00099)" — a client squarely in her book, with the exact ID supplied — came back with "You do not have access to any of the required services." And the follow-ups looped. On paper this should have been the easiest possible success: right person, right client, right ID. It failed.

4.2 The method — this is the part that impresses

The investigation, not just the fix, is the sell. Three disciplined moves:

  • Tested through the real multi-turn BFF path — not isolated gateway calls. Memory and compaction live in the BFF, so testing the gateway in isolation would have hidden the bug entirely. You have to reproduce it where it actually lives.
  • Ruled out the easy answer. The tempting conclusion is "the model is dumb." So the model was swapped gpt-4o-mini → gpt-5.4-mini across every role — and the failure was byte-identical. That single experiment proved it was not the model. A better model would not have saved it.
  • Read the glass-box trace. The trace showed intent was classified correctly (FETCH_DATA, 0.95). So the fault was downstream, in routing — not in understanding. The glass-box is what made this findable at all.

Instead of guessing, three moves: reproduce it in the real setup (not a lab shortcut that hides it); swap in a smarter brain to prove the brain wasn't the problem (same failure, so it wasn't); and read the flight recorder, which showed the machine understood the question perfectly and then mis-sent it. That narrows a vague "it's broken" down to one exact culprit.

4.3 The root cause

The semantic router embedded only the bare last message, with no domain filter, and its confidence floor was two-tier: a relative floor of topScore × 0.65 that only engaged if topScore > 0.55, otherwise it fell back to a flat absolute floor of 0.30.

Watch how those two facts combine:

  • A query like "Whitman holdings" hits the wealth agent hard — "holdings" is right there in its example prompts — scoring 0.816. Clean, confident route.
  • But "summarize for Calderon Trust" contains no domain keyword at all. Every agent scores in the noise band, around ~0.31. The top score is 0.312.
  • Because 0.312 ≤ 0.55, the relative floor never engaged. The system fell back to the porous 0.30 absolute floor.
  • That floor leaked exactly one spurious out-of-domain agent — insurance at 0.312 — while the correct wealth agents were pruned as "below-confidence-floor."
  • The segment gate then correctly denied the insurance agent (a wealth RM has no business there) → "no covered agents" → the access-denied message.

So every individual component behaved correctly. The authorization did its job. The failure was that routing handed authorization the wrong candidate to check.

The one-line insight — say this out loud in the room: the system was using its weakest signal (a fuzzy embedding of a bare sentence) and ignoring its strongest (the resolved entity's domain and the conversation context). The fix was to make it lean on the strong signal.

The receptionist was told to route only off the vague vibe of your last few words. When you said something specific-but-generic like "summarize for the Calderon Trust," it had no obvious keyword, so every desk shrugged equally — and a too-loose rule let one wrong desk squeak through. That wrong desk then got correctly turned away at security, and the whole thing reported "no access." Nothing was broken in security; the receptionist just handed security the wrong person to check. The fix: make the receptionist use what it already knows — the client's actual department and the rest of the conversation — instead of just the fuzzy vibe of one sentence.

4.4 The six fixes — all World-B-clean, all proven on wealth AND insurance

Every one of these carries zero domain literals in Java and was verified identical on two domains. Commit hashes included so you can point to the receipts.

8fce685 — Context-aware routing

Route on a conversation-enriched query (recent turns) instead of the bare message, and add margin-based abstention: route only if the top candidate decisively beats the cross-domain noise floor; otherwise CLARIFY rather than trust a noise-band hit. It reuses the manifest-fed domain filter that already existed in the Redis index. This is the fix that directly kills the root cause.

f4657e9 — Conversation focus

Three related behaviors: focal-entity (an explicit name in the current turn wins), facet-carry (a bare follow-up inherits the last data-facet — performance / settlement / cash), and bias-to-fetch (when it's genuinely uncertain whether a turn is FOLLOW_UP or FETCH, choose FETCH — never dead-end the user at "no info").

f3f2cdf — Anaphora recency

A pronoun — "their", "them", "that" — binds to the most-recently-named focal entity, and an explicit name wins the focus for the next pronoun. Proven cross-domain: as insurance underwriter uw_sam, a pronoun correctly bound to the policy just named.

570d32e — Polish (four items)

  • Out-of-coverage named entity → a clean "not in your coverage."
  • Out-of-coverage ID → copy pulled from the right domain's manifest.
  • Mixed in/out-of-segment → honest partial fulfillment: fulfill what's accessible, state what's withheld.
  • An aggregate-compute guardrail: the synthesizer refuses to fake cross-entity math — protecting the "model never computes" rule.

279bddc — Clarification composer

A manifest-declared clarify policy — a governance knob: clarify_style: template | composed. Template is deterministic and auditable (the default, plus the validated fallback). Composed is a fourth grounded LLM call site that phrases a natural, Claude-style question over the grounded candidate set (handed to it as delimited DATA) — inventing nothing, with validation (reject any ID outside the candidate set) and automatic fallback to the template.

The machine still decides to ask deterministically and still never invents anything. It just talks like a person when it asks. And the bank chooses, per department, whether it wants the strictly auditable phrasing or the warmer human phrasing.

ace3a33 — Clarify-copy

The clarification had been over-promising — telling users to "reply with the number" when it couldn't actually honor a numbered reply. It was aligned to name/identifier instead (no numbered list), and a hardcoded "relationship ID" literal was removed — itself a World-B improvement (one fewer domain string in Java).

7. The moat & anticipated client Q&A

6.1 Why it's defensible — the moat

  • Zero embedded domain knowledge — a new division is JSON, not a release.
  • LLM proposes / code disposes — fabricated IDs are structurally impossible, not merely discouraged.
  • RESOLVE is principal-agnostic; CHECK is the only gate — clean separation of "what exists" from "what you may see."
  • Partial-tolerant fan-out — degrades gracefully under real failure and says what's missing.
  • Agent output is untrusted DATA in synthesis — prompt-injection is inert by construction.
  • Glass-box observability — the reason a subtle misroute was even findable; and the disciplined process itself is part of the moat.

6.2 Tough questions the client may ask — and crisp answers

"How is this different from a chatbot?"

A chatbot answers from a model's memory. Conduit is an orchestrator: it routes your question to your real specialist systems, enforces your entitlement rules in three layers, and only ever summarizes the real data those systems return. It also carries no domain knowledge — a new business line is a config change, not a rebuild. The chatbot is the easy part; the orchestration and governance are the product.

"What stops it inventing account numbers or data?"

Structurally, the model is never allowed to produce an ID or a number. It extracts a human reference; deterministic code resolves that to a real ID against your system. If the reference doesn't resolve, there's no ID, so it asks a clarifying question rather than guessing. In synthesis, agent data is delimited input the model may only summarize — it never computes or recalls figures. Fabrication isn't policed; it's designed out.

"What happens when an agent fails?"

Its siblings still answer. We join to a deadline, harvest whatever came back, synthesize from the survivors, and the answer names what's missing. We've proven it live by killing an agent mid-answer — no hang, no silent drop, an honest "these services were unavailable."

"How do you add a new business line?"

Write a domain manifest and stand up a coverage-service URL; register each agent's manifest; if the structural rules need a new segment mapping, that's a Cerbos policy edit. No gateway code changes — and a deterministic checker enforces that the gateway stays domain-free. New division onboarding is measured in configuration, not release cycles.

"How do you add a new agent or domain, and how does the system learn to route to it?"

You drop a manifest file into the registry folder — a standard form declaring the agent's id, domain, protocol/connection, read-vs-write mode, data classification, and at least three example questions it can answer. On startup (or via one admin endpoint, live) the gateway validates it against a pinned schema, introspects its data shapes, stores it, and — the key step — embeds those example questions in-JVM and files them in a Redis HNSW vector index. From that instant, any incoming question whose meaning is nearest those examples routes to the new agent. "Learning to route" is not training or code; it's the vector index picking up the newly-embedded example prompts. Re-submitting a manifest cleanly re-indexes it. Zero gateway Java changes, and the World-B checker stays at CRITICAL 0.

"Is it secure? How is authorization enforced?"

Identity is a verified JWT (OIDC, RS256/JWKS) checked at every hop — nothing the model says is ever trusted for authorization. Every request passes three gates: segment and data-classification (structural, via the Cerbos policy engine) and coverage (data-aware, is this client actually in your book). The check is the only gate, and all three must pass.

"Why not just use a bigger, smarter model?"

We tested exactly that. Swapping the model across every role produced a byte-identical failure on the bug we found — because the fault was in routing logic, not intelligence. A smarter model still can't see your entitlements, can't reach your systems, and can't be prevented from hallucinating by raw capability alone. The value is the orchestration and the guardrails, which a bigger model doesn't provide.

"Does it scale?"

The gateway is stateless — it re-derives everything per turn, so any instance can handle any conversation; scale horizontally. Fan-out runs on virtual threads, making wide parallel dispatch cheap. And we deliberately left clean interface seams (protocol adapters, A2A stub) so the scale-version components drop in without a rewrite.

"How do you know it's right?"

Two things. Continuously, every decision emits a glass-box trace (in the UI and to OTel → Langfuse / Tempo / Loki / Prometheus) so you can audit any answer after the fact. And at release, a DeepEval gate plus committed eval fixtures must pass. The subtle misroute we fixed was findable only because of that trace — the observability is load-bearing, not decorative.

"How is a clarification not just the model guessing?"

The decision to clarify is made in deterministic gateway code — extracted entities intersected with required entities is empty — not by the model. When the optional "composed" phrasing is enabled, the model only wordsmiths the question over a grounded candidate set handed to it as data, with validation that rejects any ID outside that set and automatic fallback to a fixed template. The machine decides; it never guesses its way to an answer.

9. One-page cheat sheet — glance before you walk in

The 10-step pipeline

  1. Ingress — SSE + verified JWT; JWT is the only authz truth.
  2. Intent classify (LLM #1) — prompt compiled from manifest.
  3. Entity extract (LLM #2) — human references only, never an ID.
  4. Resolve — references → IDs via coverage svc; principal-agnostic.
  5. Clarify — deterministic: extracted ∩ required = ∅.
  6. Route — embed → Redis HNSW KNN over manifest example_prompts.
  7. Authorize — 3 gates: segment + classification (Cerbos) + coverage. All must pass.
  8. Fan-out — virtual-thread parallel; partial-result tolerant.
  9. Synthesize (LLM #3) — agent output = delimited DATA; summarize only.
  10. Egress — SSE + glass-box trace (Redis 24h) + OTel.

Memory compaction (the BFF owns it, not the gateway)

  • Why: gateway is stateless → the Chat BFF owns conversation memory; it assembles a bounded context window per turn.
  • How: ContextAssembler = recent messages VERBATIM + a rolling LlmSummaryService summary ("topics/intent only," facts-free, fire-and-forget) within a TokenCounter budget; summaryWatermark tracks what's already folded in so it rolls, not re-summarizes.
  • Guardrail: chat_compaction_summary_attached_total{attached="true"} (Grafana) — flatlines at 0 = compaction silently broke.
  • Hardening tie-in: compaction shapes what the gateway sees, so multi-turn debugging had to run through the real BFF, not isolated gateway calls.

Agent ingestion (World-B in one motion)

  • Register: agent manifest (registry/manifests/*.json) validated vs pinned agent-manifest.schema.json; domain manifest (registry/domains/*.json) for coverage URLs + copy.
  • Key fields: agent_id, domain, audience, protocol/connection, skills.examples (≥3, the routing prompts), access_mode (read/write), data_classification.
  • Pipeline: bootstrap on ApplicationReadyEvent → validate → introspect → persist (RedisJSON agent:{id}) → VectorIndex.index() embeds each example in-JVM (DJL MiniLM 384-dim) → HNSW index intent_idx (fields agent_id, domain, is_mutating, embedding).
  • Metadata = wiring: examples→routing matches; domain→route filter + segment authz; access_mode→read-only fan-out (@is_mutating:[0 0]); coverage URLs→resolve + book CHECK.
  • Onboard: drop manifest + coverage URL → restart (or POST /admin/agents) → validated, indexed, routable. No image rebuild, no gateway code, world-b 0. Re-submit = clean re-index.
