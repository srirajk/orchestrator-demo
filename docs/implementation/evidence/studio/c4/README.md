# Axiom Story C4 ‚Äî Consequence-diff review surface (the adoption unlock)

This directory is the checked-in evidence for C4: **a non-engineer approves business consequences, not
YAML.** The backend truth path computes the *decision delta* a candidate policy bundle would introduce
‚Äî as business consequences over the tenant's manifest entity vocabulary ‚Äî from **real PDP decisions**,
and binds an approval cryptographically to the exact machine consequences the human saw.

The **headline artifact is the real current-vs-candidate diff run** in this folder
([`consequence-review.json`](./consequence-review.json) + the raw PDP batches), generated by
`C4RealPdpEvidenceTest` against the **pinned Cerbos `0.53.0`** (the runtime PDP version) via an
ephemeral `docker run --rm` ‚Äî no container name, no port ‚Üí it never touches the running
`conduit-cerbos` PDP.

> **Scope note.** The React consequence-review SPA is a **separate product surface, HELD for the
> user.** C4 here delivers the *data contract* the SPA renders (a typed `ConsequenceReview`) + the
> backend truth path + the approval binding. The 6 harness tests assert on the DATA (the delta set,
> the alarm flag, the disclosure field, the review hash, staleness, binding), never on a rendered DOM.

## The problem C4 fixes

A YAML diff is unreadable to the person accountable for the risk. A domain owner cannot tell that
`scopePermissions: ‚Ä¶_OVERRIDE_PARENT` silently re-grants a denied capability. C4 shows the **decision
consequences** instead: "*N principals GAIN `<action>` access to `<entity>`*". And it makes the
approval bind to those exact consequences, so what was approved is provably what gets promoted.

## How the truth is produced ‚Äî and where the LLM is NOT

```
current bundle (immutable snapshot, bundleId)     candidate bundle (immutable snapshot, bundleId)
        ‚îÇ                                                   ‚îÇ
        ‚îî‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ real PDP evaluations over the SAME sampled fixture matrix ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò
                          (pinned Cerbos, or the in-process C3 fidelity evaluator)
                                          ‚îÇ  PdpDecisionSource
                                          ‚ñº
        diff per cell ‚Üí ConsequenceDelta[]   (DENY‚ÜíALLOW = the loud over-permission alarm)
                                          ‚ñº
        canonicalise ‚Üí consequenceReviewHash = sha256(tenant, currentBundleId, candidateBundleId,
                                                        fixtureSetHash, canonicalDelta)
                                          ‚ñº
        phrase each delta over the MANIFEST entity vocabulary        (World B ‚Äî no hardcoded domain literals)
                                          ‚ñº
        [optional] LLM phrases the finished delta into display prose  ‚Üê the ONLY LLM touch; post-hoc; never truth
```

**No LLM touches the delta, the alarm, or the hash.** The live pointer is **never** swapped to compute
a diff ‚Äî both snapshots are immutable and identified by content-addressed `bundleId`.

### How a "batch of CheckResources" is a real Cerbos evaluation

The fixture matrix is rendered as a Cerbos test suite that asserts `EFFECT_ALLOW` for every cell;
`cerbos compile -o json` evaluates it against `{base bundle + candidate child}`. Per cell, Cerbos
reports either a pass (the real decision was ALLOW) or a failure carrying `failure.actual` ‚Äî the real
decision. That `actual` is the ground-truth PDP decision. See `pdp-response-current.json`: both cells
report `RESULT_FAILED` with `"actual":"EFFECT_DENY"` under the current bundle, and pass (ALLOW) under
the candidate ‚Äî a genuine DENY‚ÜíALLOW widening.

## Reproducibility

The **6 harness tests use the in-process C3 fidelity evaluator** (`LocalPdpDecisionSource`), so the
diff behaviour is pinned byte-for-byte on any JDK 25 box with no Docker ‚Äî the same belt-and-braces
split C3 uses (constrained fidelity evaluator for the deterministic number, real Cerbos for evidence).
`C4RealPdpEvidenceTest` runs the **real pinned Cerbos** when Docker is available (`assumeTrue`,
otherwise skipped) and regenerates the files in this folder.

| Parameter | Value |
|---|---|
| Real PDP | pinned `ghcr.io/cerbos/cerbos:0.53.0` (parity with the runtime PDP), ephemeral `docker run --rm`, no name / no port |
| Reproducible PDP | `PolicyExpectationEvaluator` (the C3 constrained fidelity evaluator) |
| LLM in truth path | **none** ‚Äî an LLM may only phrase the finished delta into display prose (`ConsequenceProseModelClient`, stubbed here) |
| Review hash | `sha256(tenantId, currentBundleId, candidateBundleId, fixtureSetHash, canonicalDelta)` |
| Approval signature | HMAC-SHA256 over the approval payload; signing key from env, **no committed default** |

## Honest deferrals

- **The React consequence-review UI is HELD for the user** ‚Äî C4 here is the backend truth path + the
  typed `ConsequenceReview` data contract the SPA will render. The C4.2 DOM snapshot is deferred to
  that UI story.
- **Sampled, not formal.** The diff is an empirical measurement over a finite fixture matrix, stated
  as a first-class field on the review. It raises assurance as the matrix widens; it is never a proof.
- **`TenantActiveBundleRegistry` is an in-memory seam** here; the durable, promotion-owned active-bundle
  pointer belongs to the lifecycle story (C5).
- **Separation-of-duties on the approval** (submitter ‚âÝ approver, dual control) is specified in
  `AUTHORIZATION-APPROVAL-PROMOTION.md` and is **C5 (lifecycle / examiner-chain)** ‚Äî next, not here.
- **The live-LLM prose client** is intentionally not built; the interface + arch fence are in place so
  the live impl drops behind the same authoring-plane boundary later.
