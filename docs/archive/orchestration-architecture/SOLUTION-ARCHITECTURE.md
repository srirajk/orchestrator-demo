# Solution Architecture — Conduit multi-step orchestration (target design)

> The picture-perfect target, incorporating every decision (see DECISION-LOG) and every fix from the
> teardown. This is what we build toward. "Current" vs "target" is called out where they differ.

## 0. Thesis
One plain-English question → an LLM picks only the **goal** → deterministic code **constructs a
pipeline at runtime** from typed manifest contracts → the pipeline executes with **every edge
transforming shape, filtering entities to the user's book, and carrying the user's identity** → the LLM
returns only to phrase a **grounded** answer. **LLM at the edges; deterministic and enforced in the
middle.**

## 1. Request lifecycle (one chat turn)
```
question
  │  LLM #1: intent + entity extraction + goal signal   (never emits IDs, never draws a plan)
  ▼
ROUTE            semantic KNN over agents' example prompts → goal capability
RESOLVE          human reference → canonical ID (deterministic; principal-agnostic)
ENTITLE (admit)  structural (Cerbos: audience/segment/classification) + coverage on request-named entities
  ▼
DERIVE (resolver)  match goal.consumes.from → produces.type across registry; build DAG; topo-sort;
                   reject cycle/missing/ambiguous; RE-GATE every pulled producer (structural + coverage)
  ▼
EXECUTE (dataflow)  release each node when its deps are met (no strict layer barrier); parallel where
                    independent; per-node deadline on the critical path; partial-failure tolerant
     for each EDGE between producer→consumer:
        (a) TRANSFORM   apply the cached, schema-validated CEL/JMESPath mapping (shape)
        (b) COVER       coverage-filter any entity-typed values to the user's book (row-level)
        (c) IDENTITY    assert the user's token on the hop; agent verifies, fails closed
        (d) PREDICATE   evaluate any conditional edge (CEL) to decide release
  ▼
  LLM #2: synthesis over agent outputs as delimited DATA; forbidden from computing; numbers rendered
          deterministically; provenance hard-gate on attribution
  ▼
stream SSE
```

## 2. The type / contract model (two layers)
- **Semantic port-type (`io`)** — metadata, human-declared, decides **topology**. `consumes: {entity:<key>}`
  (leaf, from resolution, no edge) or `{from:<type>}` (edge). `produces: {name, type}`. Types are
  **namespaced + versioned** (`domain.thing@major`); major must match, minor/patch may float.
- **Wire schema** — the real fields, **introspected from the protocol** (MCP/OpenAPI/A2A), drives
  binding + validation. *(Output introspection is the key build item.)*
- **Ambiguous producers** (two producers of a type) → manifest **priority** tie-break, else clarify —
  never a silent match, never a hard refusal that kills the DAG silently.

## 3. The EDGE (the unit of enforcement + transformation)
Every producer→consumer edge does four things, in order:
1. **Transform (static half).** A per-edge mapping (CEL predicate / JMESPath or CEL object-construction
   for reshaping), **generated by the LLM and schema-validated at onboarding/first-use, cached**, keyed
   by the (producer-schema, consumer-schema) pair — never generated per request. With a **Canonical Data
   Model** this is producer→canonical→consumer (O(N) mappings). **Field-projected, never blob pass-through.**
2. **Coverage filter (dynamic half).** For any **entity-typed** value crossing the edge, run the coverage
   CHECK (against that entity's manifest-declared coverage service) and keep only the covered subset. The
   **raw principal-agnostic output never crosses un-filtered**, and the **filter is applied before any
   aggregation/count** (no set-size leak).
3. **Identity hop.** The user's token (as request-context DATA, not thread-local) is asserted on the call;
   the agent verifies and **fails closed**.
4. **Predicate (optional).** A CEL predicate over available data decides whether the edge/branch releases
   (conditional control flow).

## 4. Execution engine
- **Dataflow release, not layer barriers:** a node runs the instant its dependencies complete; fast
  siblings never wait on a slow one. Per-critical-path deadline budgeting; stream partial synthesis as
  survivors land.
- **Resilience:** per-agent bulkhead / circuit-breaker / SLA (existing harness), reused unchanged.
- **Partial-failure:** a failed/timed-out node marks its dependents unmet and skips them; survivors
  harvested; the answer states what's missing. **Completeness contract:** agents declare whether an
  output is complete or truncated; analytics **refuse to compute on incomplete input** (no wrong number
  on a partial payload).
- **Control-flow tiers:** conditional edges (CEL) · dynamic map/scatter over discovered collections
  (gated by the coverage filter) · bounded guarded loops (max-iterations + termination). Never LLM-freeform.

## 5. Enforcement model (defense in depth)
| Gate | When | What it answers |
|---|---|---|
| Structural (Cerbos) | admit + re-gate per producer | may your segment invoke this capability? |
| Coverage (book-of-business) | admit for named entities; **runtime per-edge for discovered entities** | do you cover THIS entity? |
| Per-hop identity | every agent call | is the *user* (not the gateway) asserted + verified? |
| Grounding provenance | synthesis, **hard gate** | is every stated number attributable to a source? |
Rule: **RESOLVE is principal-agnostic; the CHECK is the only gate** — now enforced at runtime on
discovered entities, not just at admission.

## 6. Two LLM calls, both at the edges
- **Front:** intent + entity extraction + goal pick. Never emits IDs, never plans.
- **Back:** synthesis — narrates grounded DATA, forbidden from computing; load-bearing numbers rendered
  by template, not the model.
- **Between:** zero LLM. Reasoning is a declared agent node. The translator/predicate expressions are
  LLM-authored **artifacts**, generated + validated + cached out of band, then deterministically run.

## 7. Request-response vs durable (one planner, two backends)
- **Request-response (build first):** in-JVM, synchronous, deadline-bounded, streamed. Plan-as-data,
  interpreted. Bounded conditionals/map fit here.
- **Durable (seam):** long-running / human-approval / durable-side-effect plans → hand the same plan to
  Temporal, return async, notify. Unbounded loops + long iteration live here. Don't build it now; leave
  the `ExecutionBackend` interface.

## 8. Observability + audit (regulator-grade)
- Emit a **`plan_graph`** event and a **fall-back reason code** on every disengage (so "answered
  multi-step" vs "silently fell to flat" is never invisible).
- Persist per request an **immutable audit record**: the resolved plan, manifest versions, routing
  scores, extraction output, coverage decisions — so a runtime-only pipeline can be **replayed and
  explained** to a regulator.

## 9. World-B invariant (constrains everything above)
The resolver, translator, coverage filter, and control-flow are **generic mechanisms over
manifest-declared contracts**. No domain names / entity literals / ID patterns / user-facing domain copy
in gateway Java. Onboard a domain = manifests + coverage URL + Cerbos mapping. `world-b-check.sh`
CRITICAL stays 0.

## 10. What is proven vs to-build (honest)
- **Proven (✅):** deterministic resolver (fuzz/oracle/mutation-tested), layered executor + partial-failure
  harvest (concurrency-oracle-tested), the whole chain live through the BFF (holdings→concentration,
  entitlement denial), World-B 0.
- **To build (the fixes, in order):** per-hop identity (F1) → coverage-as-runtime-edge-filter (F2) →
  data contract: output introspection + translator + completeness + versioned types (F3) → grounding
  provenance hard-gate (F4) → dataflow release + reason-coded fallbacks + audit record.
